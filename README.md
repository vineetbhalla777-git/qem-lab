# Quantum Error Mitigation (QEM) Framework

A complete, runnable Python framework — plus a full-stack web app on top of
it — for simulating NISQ-device noise and benchmarking six quantum error
mitigation (QEM) techniques against five standard benchmark circuits,
built on Qiskit + Qiskit Aer + Mitiq.

## 🤖 AI features: chatbot, RAG, and ML recommendation

Three additional subsystems sit on top of the core simulation framework:

### 1. RAG chatbot (bottom-right of every page)

A persistent chat panel, grounded in a knowledge base of paraphrased,
source-cited articles (`backend/rag/knowledge/*.md`) covering NISQ
fundamentals, IBM Quantum Runtime's production error mitigation options,
this project's own six techniques, and the AI concepts used here (RAG,
regression, LLMs) — plus the project's own README and REPORT. Retrieval
uses TF-IDF + cosine similarity (`backend/rag/retriever.py`) — no external
embedding API or model download required.

### 2. Optional LLM layer (chat answers + plain-English explanations)

By default, **no LLM API key is configured**, and both the chatbot and the
"Explain this result" button on the Live Experiment page work in a
retrieval/template-only fallback mode — genuinely useful, just not
LLM-composed prose. To enable full LLM-generated answers via Anthropic's
API:

```bash
export ANTHROPIC_API_KEY=your-key-here
./run_server.sh
```

`backend/llm_client.py` is the single place that checks for this key —
every caller degrades gracefully if it's absent, and the UI always tells
you which mode answered (look for the "✦ LLM-generated" vs
"⌕ Retrieval-only" / "Template-based" tag under each answer).

### 3. ML technique recommender (`/advisor` page)

A Random Forest regression model (`backend/ml/`), trained on 288 real
simulation runs this project generated by actually executing every
technique across a grid of noise strengths and repeats, predicts each
technique's expected *absolute error* at a given noise level and ranks
all six accordingly. The training data and trained model are both
committed (`backend/ml/data/training_data.csv`, `backend/ml/models/`), so
no training step is required to use it — but you can regenerate both:

```bash
python3 -m backend.ml.generate_dataset   # ~4 minutes, runs real simulations
python3 -m backend.ml.train_model        # trains + saves the model
```

The model reports honest metrics (MAE vs. a naive baseline, R²) rather
than hiding them — see `backend/ml/models/metrics.txt`. It's deliberately
scoped as "which of these six implementations wins under conditions like
these," not a universal oracle for arbitrary circuits — see the
`caveat` field the `/api/predict` endpoint always returns.

## Web app (Angular frontend + FastAPI backend)


The framework is wrapped in a live web console: a FastAPI backend that runs
real noisy-circuit simulations on demand, and an Angular frontend for
exploring it interactively.

### Quick start (uses the pre-built frontend, no Node needed)

```bash
pip install -r requirements.txt
./run_server.sh
# then open http://localhost:8000
```

The Angular app is already built and committed under
`frontend-ng/dist/`, so this is enough to run the whole site with only
Python installed.

### Frontend development mode (hot reload)

If you want to edit the Angular source and see changes live:

```bash
cd frontend-ng
npm install
npm start                 # ng serve with a proxy for /api -> localhost:8000
```

Run the backend separately (`../run_server.sh` in another terminal) so
`/api/*` calls have something to proxy to. When you're done editing, rebuild
the static bundle the backend serves:

```bash
npm run build             # outputs to dist/qem-frontend/browser
```

**Pages:**
- **Overview** — project pitch with a live animated noisy→clean signal visual
- **Live Experiment** — pick a technique, a benchmark circuit, a qubit count,
  noise level, and shot count, hit "Run," watch a real Qiskit Aer simulation
  execute server-side (~0.1–2s), and optionally click "Explain this result"
  for a plain-English readout
- **Advisor** — ML-based technique recommendation for a given noise level
- **Results** — charts + filterable table + figure gallery from the
  precomputed full benchmark sweep (`results/data/benchmark_results.csv`)
- **Techniques** — reference cards for all six mitigation methods
- **About** — the project's actual findings (what worked, what didn't, and why)
- **Chatbot** (bottom-right, on every page) — RAG-grounded Q&A about quantum
  computing, error mitigation, and this project itself

**Architecture:**
```
Browser (Angular SPA: standalone components + Router + HttpClient)
      │  fetch/XHR calls to /api/*
      ▼
FastAPI (backend/main.py)
      │  imports directly, no subprocess
      │  serves frontend-ng/dist/ + a SPA fallback route for client-side routing
      ├── backend/service.py         live simulation runs
      ├── backend/rag/               retriever.py + chat_service.py (RAG chatbot)
      ├── backend/ml/                predictor.py (trained recommendation model)
      ├── backend/explain_service.py  plain-English result explanations
      └── backend/llm_client.py      optional Anthropic API wrapper (ANTHROPIC_API_KEY)
      ▼
qem_framework/ (the simulation + mitigation library)
```

Endpoints: `GET /api/techniques`, `GET /api/benchmarks`, `POST /api/run`,
`GET /api/results`, `GET /api/figures`, `GET /api/figures/{name}`,
`POST /api/chat`, `POST /api/explain`, `POST /api/predict`.

## Selecting a benchmark circuit and qubit count

The Live Experiment page lets you choose any of the 5 benchmark circuits
(Bell, GHZ, QFT, VQE, QAOA) and, where applicable, how many qubits it uses
— not just each technique's original demo pairing. A few honest
engineering notes on how this works:

- **Each technique still pairs itself with the noise model it's actually
  designed for.** Picking a different *circuit* doesn't change MEM's
  readout-only noise or PEC's local-depolarizing noise assumption — letting
  a technique run against noise it isn't built to address would produce a
  misleading demo, not a more flexible one.
- **The default observable is chosen per benchmark family, not
  one-size-fits-all.** A naive "measure the full n-qubit Z parity string"
  default turns out to give an *exactly zero* ideal value for odd-sized GHZ
  states, and *every* Z-type observable gives zero for this project's QFT
  benchmark (its output is a uniform-magnitude superposition, so Z-basis
  correlations cancel by symmetry) — both would reintroduce the
  "percent-error blowup near a near-zero ideal value" problem this project
  already ran into once with Virtual Distillation (see REPORT.md). GHZ/Bell
  use `Z0⊗Z1` (verified ideal = +1 at every tested size); QFT uses `X` on
  its highest-index qubit (verified ideal = −1 at every tested size).
- **Two combinations are known not to work, and are disabled in the UI
  rather than left to fail silently:** PEC and CDR both fail on this
  project's QFT benchmark specifically, due to real Mitiq library
  limitations — PEC's representation step only supports 1- and 2-qubit
  gates, and CDR's training-circuit generator doesn't support the SWAP
  gates Qiskit's `QFT` instruction uses internally. `GET /api/techniques`
  reports each technique's `incompatible_benchmarks`, and the frontend
  filters the benchmark dropdown accordingly; `POST /api/run` also rejects
  an incompatible combination server-side with a clear error rather than a
  raw stack trace, in case it's called directly.
- **CDR on a fully-Clifford circuit (Bell, GHZ) still runs**, but the UI
  surfaces a caveat: CDR's near-Clifford training circuits degenerate to
  the original circuit itself when that circuit is already 100% Clifford
  gates (H, CX), so its correction is less meaningful there than on VQE,
  QAOA, or QFT.
- **Qubit counts are bounded per benchmark** (Bell is definitionally
  2 qubits; others range roughly 2–6) and, for Virtual Distillation
  specifically, capped so its 2n+1-qubit ancilla circuit never exceeds 13
  simulated qubits — keeping every live request fast.

## Circuit diagram and gate usage

Every run on the Live Experiment page also renders the actual benchmark
circuit that was simulated — a dark-themed diagram (Qiskit's `mpl` drawer,
`iqp-dark` style) plus a table of elementary gate counts. Composite library
instructions (e.g. Qiskit's `QFT`) are fully decomposed before counting, so
the numbers reflect what noise actually applies to (individual `h`, `cp`,
`swap` gates, say) rather than one opaque named block. Diagram generation
degrades gracefully — if rendering fails for any reason, the API still
returns the full numeric result with `circuit_diagram: null` rather than
failing the whole request.

## What's implemented in the framework itself



**Noise models** (`qem_framework/noise/noise_models.py`)
- Depolarizing, amplitude damping, phase damping, readout (measurement)
  error, and a "composite" model combining all of the above with realistic
  T1/T2 relaxation — each parameterized by a single scalar `strength` knob.

**Mitigation techniques** (`qem_framework/mitigation/`)
| Technique | File | Approach |
|---|---|---|
| Zero-Noise Extrapolation (ZNE) | `zne.py` | Gate folding + Richardson/linear/poly extrapolation (via Mitiq) |
| Measurement Error Mitigation (MEM) | `mem.py` | Calibration-matrix inversion (full or tensored) |
| Probabilistic Error Cancellation (PEC) | `pec.py` | Quasi-probability sampling for local depolarizing noise (via Mitiq) |
| Clifford Data Regression (CDR) | `cdr.py` | Near-Clifford training + regression (via Mitiq) |
| Dynamical Decoupling (DDD) | `dynamical_decoupling.py` | XX/YY/XYXY pulse sequences in idle windows (via Mitiq) |
| Virtual Distillation (VD, M=2) | `virtual_distillation.py` | Ancilla-assisted SWAP test, exact for arbitrary Pauli observables |

**Benchmark circuits** (`qem_framework/benchmarks/circuits.py`)
Bell state, GHZ state, Quantum Fourier Transform (QFT), a hardware-efficient
VQE ansatz (scored against a transverse-field Ising Hamiltonian), and QAOA
for MaxCut — each returns ground-truth ideal values/distributions for
scoring.

**Analysis & visualization** (`qem_framework/analysis/`)
Classical fidelity, total variation distance, error-reduction percentage,
and overhead bookkeeping, plus matplotlib plotting helpers.

## Setup

```bash
pip install -r requirements.txt
```

Tested with Python 3.12. `pip install --break-system-packages ...` may be
needed depending on your environment.

## Running the benchmark suite

```bash
python3 experiments/run_benchmark_suite.py
```

This runs all eight experiments (A–H, one per technique plus a
cross-technique comparison and a scalability study), prints a live summary
to the console, and writes:
- `results/data/benchmark_results.csv` — every numeric result
- `results/figures/*.png` — accuracy-vs-noise curves, distribution
  comparisons, technique-comparison bar charts, overhead-vs-accuracy
  scatter, and scalability plots

Total runtime is well under a minute on a single core.

## Using the framework directly

```python
from qem_framework.benchmarks.circuits import ghz_state
from qem_framework.noise.noise_models import NoiseProfile, build_noise_model
from qem_framework.utils.executors import make_expectation_executor, noiseless_expectation
from qem_framework.mitigation.zne import run_zne
from qiskit.quantum_info import SparsePauliOp

bc = ghz_state(3)
obs = SparsePauliOp.from_list([("ZZI", 1.0)])
nm = build_noise_model(NoiseProfile(strength=0.2, n_qubits=bc.n_qubits))
executor = make_expectation_executor(nm, obs, shots=4000)

ideal = noiseless_expectation(bc.circuit, obs)
result = run_zne(bc.circuit, executor, scale_factors=[1, 2, 3])
print(ideal, result.raw_value, result.mitigated_value)
```

## Important implementation note (Mitiq executors)

Files that define executor functions passed into Mitiq (`zne.py`, `pec.py`,
`cdr.py`, `dynamical_decoupling.py`, `executors.py`) deliberately **do not**
use `from __future__ import annotations`. Mitiq inspects an executor's
return-type annotation at runtime via `inspect.getfullargspec` to decide how
to parse its output; the `__future__` import turns annotations into
strings, which breaks that inspection. This is documented inline in each
file — don't add it back to those modules.
